# 역할별 하루 작업 시나리오(운영 절차)

## 공통 원칙(권장)
- 기본은 1:1, 필요 시 씬 기반 그룹챗 확장
- 합성 우선순위: **Scene > Character > Project**
- mergeMode: 기본 `override`, 일부 키만 `append`
- 외부 작가가 참여하면 “Draft → Review → Approved → Published”를 권장

---

## 1) Author(외부 작가) — “초안 작성” 하루
### 목표
새 EP(씬) 1개 + 캐릭터 1명 프롬프트/로어/룰 초안까지 만들기

### 절차
1. 프로젝트 선택: `/studio/projects` → `[projectId]`
2. 캐스트 확인: `/projects/[projectId]/cast`
3. 캐릭터 초안 작성:
   - `/cast/[characterId]/prompt`
   - 시스템 프롬프트(자아): 성격/말투/욕망 입력
   - Few-shot 대화쌍 최소 3쌍 추가
   - 캐릭터 로어북: “비밀/관점/관계” 키 정리(기본 override 권장)
4. 캐릭터 룰(필요 시):
   - `/cast/[characterId]/triggers`
   - 핵심 트리거 1~3개만(과도한 룰은 검수 리스크)
5. 씬 작성:
   - `/projects/[projectId]/scenes` → 씬 선택/생성
   - `/scenes/[sceneId]`
   - 참여 캐릭터 선택 + 그룹챗 ON/OFF
   - 씬 로어북: “이번 회차 사건” 중심(override 권장)
   - 씬 룰: 회차 전용 트리거(필요 최소)
6. Review 요청(운영 정책에 따라 UI/버튼 제공 예정)

---

## 2) Reviewer(검수자) — “승인/반려” 하루
### 목표
작가가 올린 Draft를 품질/안전/모순 관점에서 검수하고 승인/반려

### 체크리스트
- 프롬프트
  - 말투/인칭/호칭 일관성
  - Few-shot이 캐릭터 톤을 충분히 고정하는지(최소 3쌍)
- 로어북
  - 키 충돌/모순 여부
  - append 사용 키가 “누적해도 안전한지”
  - unlock(BM) SKU 유효성/표기 일관성
- 룰(트리거)
  - 조건 과잉/충돌 여부(비슷한 키워드 조건 중복)
  - 프로젝트 전역 룰과 씬/캐릭터 룰의 충돌 여부
- 씬
  - 참여 캐릭터가 의도와 맞는지
  - 씬 사건이 전역 상식/법/용어와 충돌하지 않는지

### 절차
1. 프로젝트 홈 → 캐스트/씬에서 변경사항 확인
2. 시뮬레이터로 최소 10턴 리허설(1:1 + 씬 1개)
3. 승인/반려 + 코멘트 남김(정책에 따라 UI 제공 예정)

---

## 3) Admin/Owner(운영/소유자) — “배포” 하루
### 목표
승인된 revision을 Published로 승격하여 사용자 앱에 반영

### 절차
1. Approved 상태 확인(정책에 따라)
2. 배포 단위 선택:
   - 캐릭터만 배포
   - 씬만 배포
   - 프로젝트 패키지 배포(권장: 묶음 배포는 릴리즈 단위로)
3. 배포 후 모니터링:
   - 룰 과잉으로 응답이 이상해지지 않는지
   - BM unlock이 의도대로 동작하는지

---

## 4) Editor(내부 제작) — “고도화” 하루
### 목표
전역 로어북/룰을 기반으로 캐릭터/씬의 입체감을 강화

### 절차
1. 전역 로어북(`/lorebook`)에 공통 상식/용어/법 업데이트(override)
2. 캐릭터 로어로 “관점/비밀/관계” 추가
3. 씬 로어로 “회차 사건/제약” 추가
4. append 사용은 키 단위로만 제한적으로
5. 시뮬레이터로 회차별 톤/전개 검증
